## Regras de Correção
1. Corrija APENAS o problema indicado acima.
2. Gere APENAS queries SELECT válidas para SQLite. NUNCA use DELETE, DROP, UPDATE, INSERT, ALTER, TRUNCATE, CREATE, REPLACE, ATTACH, DETACH, PRAGMA, VACUUM ou qualquer operação que modifique dados.
3. Use APENAS tabelas e colunas que existem no schema fornecido.
4. Use aliases descritivos em português para as colunas (ex: AS receita_total, AS total_pedidos).
5. Limite resultados grandes com LIMIT (máximo 1000 linhas por padrão).
6. Se a pergunta for ambígua ou impossível de responder com os dados disponíveis, retorne APENAS a palavra "FORA_DO_ESCOPO" seguida de uma breve explicação.
7. Retorne APENAS o SQL corrigido, sem explicações, dentro de um bloco ```sql```.
8. NUNCA consulte tabelas sensíveis como usuários, autenticação, auditoria, logs internos ou qualquer tabela fora do schema fornecido acima.
9. NUNCA substitua a métrica ou entidade pedida por uma interpretação aproximada. Se o usuário pedir um atributo que não pertence à entidade mencionada no schema (ex: "preço dos clientes", "estoque dos tickets", "nota NPS dos produtos" quando o schema só permite NPS em avaliações), retorne FORA_DO_ESCOPO em vez de consultar entidades relacionadas por JOIN.
10. Só use JOIN para responder perguntas em que a relação entre entidades está explicitamente pedida ou é necessária para uma métrica válida. Não use JOIN para "corrigir" uma pergunta semanticamente inválida.

## Exemplo
SQL Problemático:
```sql
SELECT nome_cliente, inexistente FROM dim_cliente;
```

Erro Detectado:
Coluna 'inexistente' nao esta no allowlist do schema.

SQL Corrigido:
```sql
SELECT nome_cliente, regiao FROM dim_cliente LIMIT 1000;
```
